docker run

Turn your image into a running container — and master the flags you'll use every day.

Where we are

From image → running container.

A container is a running instance of an image. The same image can back many independent containers — like one executable can launch many processes.

The basic command

Bare minimum. Then flags.

Docker takes the image, adds a writable layer, and runs the CMD from the Dockerfile.

docker run my-image:0.1

That's the minimum — but in practice we almost always pass a few flags. We'll walk through the seven you'll use most.

Flag · 1 of 7

Detached
-d
Run in the background.

By default, docker run attaches your terminal to the container. -d sends it to the background and returns control immediately.

Use this for long-lived services — APIs, databases, the chroma container.
Example
$ docker run -d my-image:0.1
Attached vs. detached
Default (attached) Container output streams to your terminal. Close it → container stops.
-d (detached) Runs in the background. Terminal is freed. Container keeps going.

Flag · 2 of 7

Publish a port
-p
Open a door from host → container.

Inside the container the app may listen on port 8080, but the host can't reach it until you publish.

EXPOSE in the Dockerfile is documentation. -p is what actually opens the port.
Example
$ docker run -p 8080:8080 my-image:0.1
HOST_PORT  :  CONTAINER_PORT
Host
8080
localhost:8080
Container
8080
app listens here

They don't have to match — -p 9000:8080 works too.

Flag · 3 of 7

Persistent storage
-v
Mount files into the container.

Containers are ephemeral. Volumes mount a host directory (or named volume) into the container so files persist across restarts.

Also how we mount source code into a dev container for live editing.
Example
$ docker run -v $(pwd)/data:/data my-image:0.1
Two-way bind
Host folder
$(pwd)/data
→
←
Container path
/data

Changes on either side are visible on the other instantly.

Flag · 4 of 7

Configuration
-e
Inject environment variables.

API keys, database URLs, feature flags — pass them at run time with -e. Never bake secrets into images.

Use --env-file .env to inject many at once.
Example
$ docker run -e OPENAI_API_KEY=$OPENAI_API_KEY my-image:0.1
Available inside the container
OPENAI_API_KEY=sk-…injected
CHROMA_HOST=chromadbinjected
LOG_LEVEL=infoinjected

Flag · 5 of 7

Identity
--name
A name you'll actually remember.

By default Docker assigns a random name like amazing_einstein. Give the container a real name so you can find it again.

Makes docker logs, exec, stop much friendlier.
Example
$ docker run --name rag-api my-image:0.1
Container
rag-api
--name
Without --nameamazing_einstein
With --namerag-api

Flag · 6 of 7

Cleanup
--rm
Auto-delete when the container exits.

By default, stopped containers stick around. --rm removes them automatically — perfect for one-off commands.

Keeps docker ps -a from getting cluttered.
Example
$ docker run --rm my-image:0.1 python check.py
Lifecycle with --rm
RunningContainer does its work
→
ExitsCommand finished
→
RemovedNo leftovers

Flag · 7 of 7

Interactive
-it
Drop into a shell inside the container.

-i keeps stdin open, -t allocates a TTY. Together they give you a real interactive terminal.

Indispensable for exploring an image's filesystem and debugging.
Example
$ docker run -it --rm my-image:0.1 bash
What you get
root@7a9f1c2:/app# ls
main.py requirements.txt
root@7a9f1c2:/app# python -V
Python 3.11.10
root@7a9f1c2:/app# █

Putting it all together

A realistic run command.

Detached, named, with a published port, mounted volume, and injected env var.

docker run -ddetached
  --name rag-apifriendly name
  -p 8080:8080publish port
  -v $(pwd)/data:/datamount volume
  -e OPENAI_API_KEY=$OPENAI_API_KEYinject env
  rag-api:0.1image to run

Once it's running

A few commands to manage the container.

These cover the day-to-day operational side of Docker.

docker psWhat's running?Lists every active container.
docker logs <name>What did it say?Prints stdout/stderr. Add -f to follow.
docker exec -it <name> bashGet insideOpen a shell in a running container.
docker stop <name>Shut it downSIGTERM, then SIGKILL after a grace period.
docker run + a few flags = a controlled, observable container.
