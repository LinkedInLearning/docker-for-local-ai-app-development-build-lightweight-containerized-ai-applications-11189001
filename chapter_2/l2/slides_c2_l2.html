Anatomy

Top to bottom, every time.

Docker reads each instruction in order and produces a new layer for most of them. Almost every Dockerfile follows the same shape:

1
Start from a base imageFROM …
2
Define build-time arguments and env varsARG, ENV
3
Set a working directoryWORKDIR /app
4
Install system packagesRUN apt-get install …
5
Copy application filesCOPY . .
6
Install application dependenciesRUN pip install …
7
Declare the runtime commandCMD ["python", "main.py"]

Instruction · 1 of 7

Base image
FROM
Picks the starting layer your image is built on.
  • The first non-comment line of almost every Dockerfile.
  • Pin a specific tag (:3.11-slim) — never use latest.
  • Determines OS, language runtime, default tools.
Example
FROM python:3.11-slim
Choose your starting point
python:3.11-slim ✓
ubuntu:22.04
node:20-alpine
scratch  — empty

Instruction · 2 of 7

Build-time command
RUN
Executes a command during the build, baking the result into a new layer.
  • Install packages, run setup scripts, fetch data.
  • Each RUN typically adds one layer.
  • Combine with && and clean up in the same line.
Example
RUN apt-get update \
    && apt-get install -y curl \
    && rm -rf /var/lib/apt/lists/*

→

Instruction · 3 of 7

Bring files in
COPY
Copies files from your project folder into the image.
  • Source paths are relative to the build context.
  • Exclude noise with a .dockerignore file.
  • Prefer COPY over ADD — fewer surprises.
Example
COPY requirements.txt /app/
COPY src/             /app/src/

Instruction · 4 of 7

Current directory
WORKDIR
Sets the current directory for every instruction that follows.
  • Think of it as a persistent cd command.
  • Auto-creates the directory if it doesn't exist.
  • Use it instead of RUN cd /app && ….
Example
WORKDIR /app

# All later instructions run inside /app
COPY . .
RUN  pip install -r requirements.txt
Where you are inside the image
/ › app ▍

Instruction · 5 of 7

Variables
ENV / ARG
Two flavors of variables: one for the build, one for runtime.
  • ARG — only during docker build. Override with --build-arg.
  • ENV — persists into the running container. Override with -e.
  • Often paired: ARG at build, then ENV X=$X for runtime.
Example
ARG PYTHON_VER="3.11"
ENV PYTHON_VER=$PYTHON_VER
Lifespan
ARG
build-time only
Visible during docker build. Gone afterwards.
build → end of build
ENV
build + runtime
Visible to processes inside the running container.
build → runtime

Instruction · 6 of 7

Documentation
EXPOSE
Documents the port the application listens on.
  • Does not publish the port — that's a runtime flag.
  • Pure documentation for consumers of the image.
  • The actual publish happens at docker run -p.
Example
EXPOSE 8080
Like a sticker on the image
image 8080
Documentation only. Lets the next person know which port to publish.

Instruction · 7 of 7

Startup
CMD
The default command that runs when the container starts.
  • Overridable by passing args to docker run.
  • Use the exec form — JSON array — for correct signal handling.
  • Combine with ENTRYPOINT for fixed cmd + default args.
Example
CMD ["python", "main.py"]
What runs when the container starts
PID 1
python main.py
running container

All together

Seven lines. A reproducible recipe.

Here's how the core instructions fit together for a minimal Python app.

1FROM python:3.11-slim Base image
2WORKDIR /app Working dir
3COPY requirements.txt . Deps first
4RUN pip install --no-cache-dir -r requirements.txt Install
5COPY . . Then source
6EXPOSE 8080 Port
7CMD ["python", "main.py"] Run

Notice the order

Requirements before the rest of the code.

Docker caches each layer. As long as requirements.txt doesn't change, the expensive pip install step is skipped on the next build.

✗ Without thought
COPY . .                # changes every edit
RUN  pip install ...    # rebuilt every time
Any code change invalidates the install layer. Slow.
✓ Cache-friendly
COPY requirements.txt .  # changes rarely
RUN  pip install ...     # cached
COPY . .                 # only this is rebuilt
Install layer stays cached. Fast iterations.

We'll cover this in more depth in Lesson 5 — Dockerfile best practices.

Takeaway

A Dockerfile is just a short, ordered list.

It describes how to build the environment your application needs.

FROM → WORKDIR → COPY → RUN → EXPOSE → CMD
