Definition

What a registry is.

A service that stores and distributes container images. You've been using one all along.

Every FROM python:3.11-slim quietly pulls that base image — by default from Docker Hub, the registry Docker uses unless told otherwise.

Step · pull

Pulling, made explicit.

The build pulls base images for you, but you can also fetch one directly — downloading the image and its layers into your local store, ready to use.

Anatomy

How images are named.

A full reference has four parts. Type a short name and Docker fills in the defaults — but to push your own image, the namespace must be your username.

docker.io/library/python:3.11-slim
registryDefaults to docker.io.
namespacelibrary = official images.
repoThe image name itself.
tagThe version label.

To push your own image, swap the namespace for your Docker Hub username: myuser/demo:0.1.

The workflow

Sharing is three steps.

Log in, tag with your namespace, push. Each step does exactly one thing.

1
Log in
Authenticate to Docker Hub. Prompts for your username and a password or access token, then stores the credentials locally.
docker login
2
Tag
Add a second name under your namespace. This does not copy the image — both names point at the same one.
docker tag demo:0.1 myuser/demo:0.1
3
Push
Upload the image. Docker sends only the layers the registry doesn't already have — the same caching idea, now over the network.
docker push myuser/demo:0.1

Step 2 · tag

Tagging adds a name.

docker tag doesn't duplicate anything. It gives the existing image a second reference — one a registry will accept.

After tagging, demo:0.1 and myuser/demo:0.1 are the same image, same ID, just two names.
One image, two references
local name
demo:0.1
→

The command
$ docker tag demo:0.1 myuser/demo:0.1

Same IMAGE ID in docker images — only the repository column differs.

Step 3 · push

Pushing uploads only what's new.

Docker uploads each layer the registry doesn't have yet. Layers it already knows are skipped — caching, over the network.

$ docker push myuser/demo:0.1

The round trip

Pull it back anywhere.

The image now lives on Docker Hub. On any other machine — a colleague's laptop, a server, a CI runner — it's one command away.

# on a colleague's laptop, a server, or a CI runner
$ docker pull myuser/demo:0.1
0.1: Pulling from myuser/demo
a2318d6c47ec: Pull complete
b1c2d3e4f5a6: Pull complete
Status: Downloaded newer image for myuser/demo:0.1

Push here, pull there — that round trip is how an image travels from your machine to where it actually runs.

Before we move on

Two things worth knowing.

Who can pull it — and why a missing tag is a trap.

Public vs. private

You choose visibility per repository on Docker Hub.

public · anyone pulls private · auth required

The latest trap

Push without a version — just myuser/demo — and Docker assumes :latest. That moving tag silently changes out from under you.

Tagging strategy returns in Chapter 5, when we prepare images for production.

Takeaway

The everyday registry loop.

Pull base images, tag your own with your namespace, push to share, pull to deploy.

pull
