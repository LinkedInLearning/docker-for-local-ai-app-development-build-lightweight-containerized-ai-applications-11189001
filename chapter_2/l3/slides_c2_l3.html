The basic syntax.

Three things are happening — let's name each one.

Concept · 1

The build context.

The folder Docker sends to the build engine. Every file COPY or ADD can reach must live inside it.

A large context = slow build + bigger images. Trim it with a .dockerignore.
.dockerignore filters what gets sent
Project folder
Dockerfile send
requirements.txt send
main.py send
.git/ skip
__pycache__/ skip
.env skip
→
Build engine
Dockerfile
requirements.txt
main.py
~5 KB instead of 500 MB.

Concept · 2

Tag the image.

The -t flag gives the image a name and an optional version. Without a tag, the image is hard to refer to later.

Omit the version and Docker assumes :latest — which is almost never what you want.
Anatomy of a tag
my-image : 0.1
name · colon · version
rag-api:0.1
rag-dev:dev
my-app:2025-05-28

Concept · 3

Point to a Dockerfile.

By default docker build looks for a file named Dockerfile in the context. Use -f when you have a custom name.

-f only changes which Dockerfile to read. The build context (the .) is independent.
Example
$ docker build -f docker/Dockerfile_API -t rag-api:0.1 .
Which file gets built
○ docker/Dockerfile default
● docker/Dockerfile_API selected via -f
○ docker/Dockerfile_Dev unused

What docker build does

Each instruction → one layer.

Docker reads the Dockerfile top-to-bottom, creating a new read-only layer for most instructions. The final stack is the image.

FROM python:3.11-slim
COPY requirements.txt .
RUN pip install ...
COPY . .
CMD [...]
→
Image · stack of layers
⚙ metadata (CMD)
application source
installed packages
requirements.txt
python:3.11-slim (base)

What makes rebuilds fast

Layers drive the build cache.

On the second build, Docker reuses layers whose inputs haven't changed — and invalidates everything below the first change.

First build · everything fresh
⚙ FROM python:3.11-slim
⚙ COPY requirements.txt .
⚙ RUN pip install ...
⚙ COPY . .
⚙ CMD [...]
After editing main.py
✓ FROM (cached)
✓ COPY requirements.txt . (cached)
✓ RUN pip install ... (cached)
✗ COPY . . (changed → rebuilt)
✗ CMD [...] (rebuilt)

Order matters — that's why COPY requirements.txt comes before COPY . .

Parameterize the build

Pass build args from the CLI.

--build-arg overrides any ARG declared in the Dockerfile — perfect for tool versions or feature flags.

Our project uses this to set the Python version from the build script.
Example flow
1
Declare in Dockerfile: ARG PYTHON_VER="3.11"
↓
2
Override at build: --build-arg PYTHON_VER=3.12
↓
3
Used inside Dockerfile: RUN install_uv.sh $PYTHON_VER
Full command
$ docker build --build-arg PYTHON_VER=3.12 \
    -t my-image:0.1 .

Useful flags

A few more worth knowing.

These come up often when debugging builds or shipping multi-arch images.

--no-cache Ignore the cache Rebuild every layer from scratch — useful when you suspect a stale cache entry.
--progress=plain Full log output Show the full, unwrapped output of each step instead of the compact view.
--platform Target a specific architecture Build for linux/amd64, linux/arm64, or both with buildx.
--pull Refresh the base image Always re-pull FROM rather than trust the local cache.
