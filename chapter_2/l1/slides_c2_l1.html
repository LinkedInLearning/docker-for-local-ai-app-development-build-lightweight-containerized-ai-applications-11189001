The workflow

Four steps. Every time.

Each step has a clear role and a clear output. We'll walk through each one in turn.

1
Think

Requirements

List what the app needs to run — runtime, libs, env, ports.

2
Write

Dockerfile

A plain-text recipe that captures those requirements.

3
Build

docker build

Turn the recipe into a reusable, immutable image.

4
Run

docker run

Launch a container — a running instance of the image.

Step 1 of 4

1

Start from clear requirements.

The most-skipped step — and the one that decides whether your Dockerfile stays clean.

"The clearer the requirements, the cleaner the Dockerfile."

Step 2 of 4

2

The Dockerfile is the recipe.

A plain-text file that translates the requirements into instructions Docker can follow, one per line.

The Dockerfile is the recipe.
The image is the cake.
1# A minimal Python image
2FROM python:3.11-slim Base image
3WORKDIR /app Where things live
4COPY requirements.txt . Deps first
5RUN pip install --no-cache-dir -r requirements.txt Install
6COPY . . Then source
7EXPOSE 8080 Port
8CMD ["python", "main.py"] How to run

Seven lines — a fully reproducible recipe for the application's environment.

Step 3 of 4

3

docker build creates the image.

Docker reads the Dockerfile top-to-bottom and creates a new layer for most instructions.

Images are immutable. Change something → build a new image.
Each instruction → one layer in the image

COPY requirements.txt .
RUN pip install ...
COPY . .
CMD [...]

Step 4 of 4

4

docker run starts a container.

The image is a template. A container is a running instance — with its own filesystem, network interface, and isolated processes.

One image → many independent containers.
Like one .exe → many running processes.

In practice

It's a loop, not a line.

Every code or dependency change walks through the workflow again — but the build cache keeps repeat passes fast.

Editcode or requirements
docker buildrebuild changed layers
docker runstart a fresh container
