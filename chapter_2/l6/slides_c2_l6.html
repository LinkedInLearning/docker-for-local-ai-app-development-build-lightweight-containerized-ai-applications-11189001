The map

Docker manages four kinds of objects.

Each one has the same little family of subcommands: ls, inspect, rm, prune. This lesson focuses on the two you touch every day.

focus

Images

Read-only templates you built or pulled.

docker images
docker image inspect
docker rmi / image prune
focus

Containers

Running or stopped instances of an image.

docker ps / ps -a
docker logs / exec / inspect
docker rm / container prune

Volumes

Persistent storage attached to containers.

docker volume ls
docker volume inspect / rm
docker volume prune

Networks

Virtual networks containers connect to.

docker network ls
docker network inspect / rm
docker network prune

Volumes and networks come back in Chapter 3, when the RAG project gets multi-service.

Step 1 · what's on the box?

Start with docker ps and docker images.

A snapshot of what's on your machine right now. By default, docker ps hides stopped containers — add -a to see them all.

$ docker ps
CONTAINER ID IMAGE COMMAND STATUS PORTS NAMES
3b9f1c2a4d5e demo:0.1 "python main.py" Up 12 minutes 0.0.0.0:8080->8080/tcp demo
9a8b7c6d5e4f chromadb:1.3 "chroma run ..." Up 12 minutes 0.0.0.0:8000->8000/tcp chromadb
$ docker ps -a # includes stopped
CONTAINER ID IMAGE COMMAND STATUS NAMES
3b9f1c2a4d5e demo:0.1 "python main.py" Up 12 minutes demo
e5d4c3b2a1f0 demo:0.0.9 "python main.py" Exited (0) 2 hours ago old-demo
$ docker images
REPOSITORY TAG IMAGE ID CREATED SIZE
demo 0.1 7c4e8a1b9f23 3 minutes ago 145MB
python 3.11 a1b2c3d4e5f6 2 weeks ago 1.02GB
<none> <none> e5d4c3b2a1f0 1 hour ago 420MB <-- dangling

A <none>:<none> entry is a dangling image — an old build replaced by a newer one. Safe to prune.

Step 2 · what is it doing?

Three commands cover 95% of debugging.

When something looks wrong, ask: what's it saying, what's inside, what's its config?

docker logs

Print everything the container wrote to stdout/stderr. Add -f to follow live.

docker logs -f demo
docker logs --tail 100 demo
docker logs --since 10m demo
docker exec

Run a command inside a running container. The single most useful debugging tool.

docker exec -it demo bash
docker exec demo ls /app
docker exec demo env
docker inspect

Dump every detail Docker knows: mounts, env, network, command, restart policy.

docker inspect demo
docker inspect demo \
  | jq '.[0].Config.Env'

Bonus: docker stats for live CPU/memory, docker top for processes inside.

Step 3 · control its life

The container lifecycle.

Containers move through a small set of states. A handful of commands take you from one to the next.

State machine
commandwhat it does
docker stopSIGTERM, then SIGKILL after a grace period.
docker startRestart a stopped container with its original config.
docker restartStop + start in one step.
docker killImmediate SIGKILL — no grace.
docker rmRemove a stopped container. Add -f to force.
docker rmiRemove an image. Refuses if any container uses it.

Step 4 · disk usage

Builds and runs add up. Fast.

After a few weeks of active development, 50+ GB of unused Docker data on disk is normal. The first command to know is docker system df.

$ docker system df
Images
18.2 GB · 77%
Containers
142 MB · 90%
Volumes
3.4 GB · 61%
Build cache
9.7 GB · 100%
bars show reclaimable portion

The percentages tell you how much can be cleaned up. The next slide is the prune cascade — from safe to nuclear.

Cleaning up

From safe to nuclear.

Targeted prunes first. docker system prune -a --volumes last — powerful, but irreversible.

✓ Safe · targeted
docker container prune
docker image prune
docker volume prune
docker network prune
Each removes only one object family. Prompts for confirmation; add -f to skip.
! Aggressive
docker image prune -a
docker system prune
docker builder prune
image prune -a removes every image not used by a container. system prune sweeps containers, networks, and dangling images.
⚠ Nuclear
docker system prune -a
docker system prune -a --volumes
--volumes deletes named volumes too — including chroma_data if its container is down. Use with care.

Rule of thumb: re-run docker system df before and after to see what was reclaimed.

The commands worth memorizing.

The ones you'll reach for every day. Full reference and sample output is in the lesson README.

QuestionCommand
