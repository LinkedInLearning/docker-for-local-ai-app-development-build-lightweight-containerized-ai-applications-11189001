1
Best practice · 1 of 7

Pin every version.

A Dockerfile that starts with FROM python may build today and break tomorrow when latest moves. Reproducibility starts with pinning.

✗ Bad
FROM python
RUN apt-get install -y curl
RUN pip install fastapi
Floating latest, no version pins. Will silently change.
unstable · drifts over time
✓ Good
FROM python:3.11-slim
RUN apt-get update \
    && apt-get install -y --no-install-recommends curl=7.88.* \
    && rm -rf /var/lib/apt/lists/*
RUN pip install --no-cache-dir fastapi==0.115.0
For full reproducibility, pin by digest: python:3.11-slim@sha256:…
reproducible · same build every time
2
Best practice · 2 of 7

Order instructions for cache efficiency.

Docker caches each layer. The moment one instruction changes, every layer below it is rebuilt. Put stable steps first, volatile last.

✗ Code first, install later
FROM python:3.11-slim
WORKDIR /app
COPY . .                # changes constantly
RUN pip install -r requirements.txt
Every code edit invalidates the install layer.
pip install on every build
✓ Deps first, code last
FROM python:3.11-slim
WORKDIR /app
COPY requirements.txt .   # changes rarely
RUN pip install -r requirements.txt
COPY . .                # only this rebuilds
Dependency layer stays cached across code edits.
cached · fast iterations

Best practice · 3 of 7

Use ARG and ENV deliberately.

ARG for build-time knobs. ENV for runtime config. Use both when a value is set at build but must be visible at runtime.

▸ Build-time pattern
ARG PYTHON_VER="3.11"
ARG VENV_NAME="my_project"

ENV PYTHON_VER=$PYTHON_VER
ENV VENV_NAME=$VENV_NAME

COPY install.sh settings/
RUN  bash ./settings/install.sh \
        $VENV_NAME $PYTHON_VER
Override at build with --build-arg PYTHON_VER=3.12 without editing the file.
▸ When to use which
Use case ARG ENV
Tool / runtime version✓opt
Build feature flag✓✗
Runtime config (paths)✗✓
Runtime secret✗via -e

Best practice · 4 of 7

One RUN, not five.

Every RUN creates a layer. Split installs balloon the image — and cleanup in a later layer doesn't actually shrink it.

✗ Three layers, cache stays
RUN apt-get update
RUN apt-get install -y curl git
RUN rm -rf /var/lib/apt/lists/*
Apt cache is baked into the previous layer — cleanup in a new layer doesn't help.
+3 layers · larger image
✓ One layer, cleaned up
RUN apt-get update \
    && apt-get install -y --no-install-recommends curl git \
    && rm -rf /var/lib/apt/lists/*
--no-install-recommends often halves the size further.
+1 layer · smaller, cleaner
5
Best practice · 5 of 7

Move complex setup into shell scripts.

When a RUN grows past a few lines, extract it. Easier to read, easier to test in isolation.

✗ 30-line RUN inside Dockerfile
RUN apt-get update \
    && apt-get install -y curl git zsh \
       fzf btop bat unzip jq \
       build-essential locales \
    && sh -c "$(curl ...oh-my-zsh...)" \
    && chsh -s $(which zsh) \
    && gem install colorls \
    && locale-gen en_US.UTF-8 \
    && # … and 20 more lines
Hard to read. Hard to test. Tightly coupled to the Dockerfile.
✓ Externalized into a script
COPY install_dependencies.sh settings/
RUN  bash ./settings/install_dependencies.sh
Exactly how docker/Dockerfile_Base works in this repo. The Dockerfile stays a high-level outline.
readable · testable

Best practice · 6 of 7

Use a .dockerignore. Always.

Caches, secrets, and version-control noise have no business in your image. Excluding them shrinks the image, speeds up the build, and improves cache hits.

.dockerignore
.git
__pycache__/
*.pyc
.venv/
node_modules/
.env
.env.*
*.log
.DS_Store
.pytest_cache/
.ruff_cache/
What ends up in the context
main.pysent
requirements.txtsent
src/sent
.git/skip
__pycache__/skip
.envskip
.venv/skip

5 KB  vs  500 MB

Best practice · 7 of 7

Drop root.

Production containers should not run as root. Create a dedicated user and switch to it before declaring the runtime command.

✓ Switch to a non-root user
RUN useradd -m -u 1000 -s /bin/bash appuser \
    && chown -R appuser:appuser /app
USER appuser

EXPOSE 8080
CMD ["python", "main.py"]
Use a fixed UID/GID so the user ID is predictable when bind-mounting host directories.
▸ Who runs the process?
Default: root Risky in production
uid=0
With USER: appuser Safe by default
uid=1000

For dev containers, root is often acceptable — make it a conscious choice.

Bonus

A few smaller habits.

Quick wins that compound across your Dockerfiles.

Prefer COPY over ADDADD has hidden behaviors (URL fetch, tar auto-extract). Use it only when you need them.
Use the exec formCMD ["python", "main.py"], not CMD python main.py. Signal handling works correctly.
Add a HEALTHCHECKLets Docker and orchestrators know when the app is actually ready to serve traffic.
Add OCI LABELsSearchable metadata: title, version, source. Easy to inspect with docker inspect.
Set Python env varsPYTHONDONTWRITEBYTECODE=1 and PYTHONUNBUFFERED=1 — cleaner image, real-time logs.
Multi-stage for compiled langsBuild in a fat image, copy artifacts into a slim one. Toolchain never ships.
