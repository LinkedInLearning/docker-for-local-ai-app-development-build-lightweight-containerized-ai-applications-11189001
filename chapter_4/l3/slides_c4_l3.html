The stack

Three services on one network.

docker-compose.test.yaml
services:
  ingestion:
    build: { dockerfile: …/Dockerfile_Ingestion }
    ports: ["8081:8081"]
    depends_on: { chromadb: { condition: service_healthy } }
  query:
    build: { dockerfile: …/Dockerfile_Query }
    ports: ["8080:8080"]
  chromadb:
    image: chromadb/chroma:1.3.5
    healthcheck: { test: python …/heartbeat }
networks: { rag-net: { driver: bridge } }
What to notice
buildCompose builds the two app images from the Lesson 2 Dockerfiles — it builds as well as runs.
healthcheckThe database reports whether it is ready, not just started.
depends_onApp services wait for the DB to be service_healthy.
rag-netOne bridge network; each service reachable by name.

Why a healthcheck

depends_on waits for start, not ready.

A database is "started" a second or two before it accepts connections. The healthcheck closes that gap.

× depends_on alone
DB container started
ingestion starts immediately
DB not yet listening → connection refused
Waits for the container to start — not for the service inside to be ready.
✓ condition: service_healthy
DB container started
heartbeat check passes → healthy
only then do app services start
Compose asks the DB "are you ready?" and holds the apps back until its heartbeat passes.

Hands-on · terminal

Let's bring the stack up.

From the project root, one command builds both app images, starts the database, waits for its healthcheck, then starts ingestion and query. Switch to the terminal.

up -d --build ps logs query down -v

Terminal · build & bring it up

One command, three containers.

$ docker compose -f chapter_4/l3/docker-compose.test.yaml up -d --build

✓ Container l3-ingestion-1 Started
✓ Container l3-query-1 Started
