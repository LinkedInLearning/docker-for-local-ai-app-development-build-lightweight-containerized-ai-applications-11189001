Best practice · 2 of 4

Health checks, everywhere.

We added a healthcheck to the database in Lesson 3. The same idea belongs on every service — a HEALTHCHECK hitting /health. 

HEALTHCHECK --interval=30s --timeout=3s \
  CMD curl -f http://localhost:8080/health || exit 1
Gates startupCompose waits until a dependency is healthy before starting what needs it.
Signals readinessAn orchestrator uses it to decide a container is ready for traffic.
Pinpoints failuresTells you which service is down when something breaks.
3
Best practice · 3 of 4

Manage your test data.

Integration tests need data, and that data must be predictable.

1
Small fixed sample

Reuse one tiny pdf/ document — not a 500-page report that makes every run slow.

2
Ephemeral per run

A fresh database collection each run, so runs never contaminate each other.

3
Tear it down

docker compose down -v removes the volumes — the next run starts clean.

Goal — reproducibility: the same test, run twice, gives the same result.
4
Best practice · 4 of 4

Run the whole stack in CI.

Up
bring the stack up
Test
run tests against it
Down
tear it down
# .github/workflows/test.yaml
- run: docker compose -f chapter_4/l3/docker-compose.test.yaml up -d --build
- run: pytest chapter_4/l4/test_integration.py
- run: docker compose -f chapter_4/l3/docker-compose.test.yaml down -v

The stack is a Compose file and the images are pinned — so the environment in CI is the same one you ran locally, and close to production. Every change gets tested against the real multi-container topology, automatically.

The gap

“Close to production” is not production.

We have a tested, cooperating multi-container app — but three things still stand between it and a real deployment.

Images not optimizedBuilt for development, not for size or security.
Nothing deployedEverything still runs on your laptop — not a server or the cloud.
No scaling or observabilityNo replicas, no metrics, no real production monitoring.
