The principle

One responsibility per container.

A container should do one job and do it well. When we cram the whole application into one container, we lose the things containers are good at.

We also tie together parts of the system that have nothing in common — a bursty batch pipeline bolted to a latency-sensitive request handler.

Why split

Five reasons to split.

Independent scaling

The two pipelines have opposite load profiles — scale them apart.

Ingestionbursty · CPU-heavy · batch
Querysteady · latency-sensitive

Failure isolation

A giant document ties up ingestion; query keeps answering.

Independent deploy

Fix the query prompt and redeploy without rebuilding the parser.

Right-sized images

Each image carries only what its service needs — faster, safer.

Clear boundaries

Separate secrets, network exposure, and resource limits.

Testing near production

Where the real bugs live.

One process

Everything can reach everything — for free.

  • Same memory space
  • No hostnames, ports, or access rules
  • Hides the integration bugs
Separate containers

Services must talk over a network.

  • Hostnames resolve to the right service
  • Ports are published correctly
  • Access rules are enforced
Run the topology now — surface the bugs while they're cheap to fix.

Takeaway

One job per container.

Split along the seam the prototype already drew, then test the services together over a network.
