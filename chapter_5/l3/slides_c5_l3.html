The shape of image security

Three habits, not one switch.

Security in a container image comes down to three repeatable habits — each one shrinks how much can go wrong.

Reduce surface
Less in the image means less to attack — minimal base, drop the extras.
Least privilege
Run as non-root, read-only, no capabilities you don't need.
Scan for CVEs
Find known vulnerabilities, fix them, and prove they're gone.

Habit 1 · least privilege

Don't run as root.

By default a container process is root. A process that escapes a root container is root on resources it should never reach. Create an unprivileged user and switch to it.

Dockerfile
# create a non-root user, then switch
RUN useradd --create-home --uid 10001 appuser
USER appuser
What it buys you
USEREvery later layer and the runtime process run as appuser, never root.
--uidA fixed, high UID — predictable for volume permissions and host mappings.
biggest winCheapest, highest-impact hardening you can apply to an image.

Habit 1 · reduce surface

Pin the base. Keep it minimal.

python:3.11-slim floats — the tag moves under you. Pin by digest for a reproducible, verifiable build.

Dockerfile
# floating tag — moves under you
FROM python:3.11-slim

# pinned by digest — reproducible
FROM python:3.11-slim@sha256:<digest>
Why it matters
@sha256The exact bytes you tested are the exact bytes you ship — no silent drift.
minimalPrefer slim or distroless — fewer packages, fewer CVEs.
verifiableAnyone can confirm the base is exactly what you claim.

Habit 2 · least privilege

Keep secrets out of the layers.

API keys and tokens must never be baked into a layer — layers are cached, shared, and pushed. Inject them at runtime, then verify nothing leaked.

# inspect every layer's build command — no keys should appear
$ docker history --no-trunc rag-query:0.1.0
CREATED BY
RUN useradd --create-home --uid 10001 appuser
COPY . /app
CMD ["uvicorn", "app.main:app", ...]
# no OPENAI_API_KEY, no tokens, no .env baked in
# keys arrive at runtime: docker run -e OPENAI_API_KEY=$OPENAI_API_KEY …

Habit 1 + 2 · reduce surface

Drop what you don't need.

Trim at build time, then constrain at runtime — the process gets only the privileges it actually uses.

Build time — Dockerfile
RUN apt-get update \
 && apt-get install --no-install-recommends \
    -y curl \
 && rm -rf /var/lib/apt/lists/*
Runtime — docker run
--read-onlyRoot filesystem is immutable — nothing can be written into the running image.
--cap-drop ALLStrip every Linux capability; add back only what the app truly needs.
same layerClean apt lists in the same RUN so they never persist in a layer.
$ docker run --read-only --cap-drop ALL rag-query:0.1.0

Habit 3 · scan

Scan for known CVEs.

A scanner reads your image's packages and reports known vulnerabilities — by severity, with the fixed version where one exists.

$ docker scout cves rag-query:0.1.0
# or: trivy image --severity HIGH,CRITICAL rag-query:0.1.0
Target rag-query:0.1.0
CRITICAL CVE-2024-XXXX libssl 3.0.2 → fixed in 3.0.13
HIGH CVE-2024-YYYY zlib 1.2.11 → fixed in 1.2.13
1 CRITICAL, 1 HIGH — action required

Habit 3 · the loop

Scan, fix, re-scan.

A scan is only useful if you act on it. The loop — not any single scan — is the habit.

Scan
Find the CVE
Run the scanner; read the offending package and its fixed version.
Fix
Bump & rebuild
Move the base image or dependency to a fixed version, then rebuild.
Re-scan
Prove it's gone
Run the scanner again — the CVE should no longer appear.
Repeat until clean. Scan → fix → re-scan is the security habit you keep.

For AI workloads

Never bake keys or weights.

The AI-specific trap: secrets and model artifacts end up frozen into a shared, pushable layer.

× baked into the image
ENV OPENAI_API_KEY=… in the Dockerfile
COPY model.gguf — weights as a layer
docker history → key & weights leak
Layers are cached, shared, and pushed — anything baked in travels with the image.
✓ injected as data at runtime
docker run -e OPENAI_API_KEY=…
mount or download weights as data
docker history → nothing sensitive
The image stays a clean, shareable artifact; secrets and weights stay outside it.

Takeaway

Smaller and hardened.

The image now runs non-root, pins a minimal base, carries no secrets, and passes a clean scan — and you keep it that way with the scan → fix → re-scan loop.
