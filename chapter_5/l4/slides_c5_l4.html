The problem

You develop on one CPU, deploy on another.

Apple Silicon laptops are arm64; much of the cloud is amd64; some cloud is ARM. The image has to match where it runs.

× amd64-only image on an arm64 host
image built for linux/amd64 only
won't start on arm64 — or silently runs under emulation
emulated workload crawls
Build for the wrong architecture and you find out in production.
✓ build for every target
one Dockerfile, named platforms
a native binary per architecture
each host runs the image made for it
The fix is a build that produces more than one architecture.

The tool

Buildx + a docker-container builder.

Buildx is Docker's interface to the BuildKit engine. The default docker builder targets only your current arch; a docker-container builder targets many.

$ docker buildx create --name multiarch --driver docker-container --use --bootstrap
[+] Building... ✓ multiarch booted
$ docker buildx inspect --bootstrap # lists the platforms it can target
Name: multiarch
Driver: docker-container
Platforms: linux/amd64, linux/arm64, linux/arm/v7, linux/386 ...

The build

One source, many binaries.

name the platforms
# one Dockerfile, two architectures
docker buildx build \
  --platform linux/amd64,linux/arm64 \
  -t rag-demo:0.1.0 .

The same build command — you just list the targets after --platform.

what changed
--platformA comma-separated list of targets. Buildx compiles the image once per architecture.
one DockerfileThe exact same source produces every binary — no per-arch forks.
the resultOne build, multiple images: amd64 and arm64 from a single run.

How it builds other arches

Emulation is convenient; native is fast.

How does an amd64 machine build arm64? It emulates the foreign CPU with QEMU, bundled with Docker Desktop.

QEMU emulationdefault · Docker Desktop
Native buildersspeed at scale
Translates foreign-arch instructions on the fly. Works anywhere, no extra hardware — but the foreign-arch build runs slowly.
A real machine of each architecture builds its own image at full speed. More setup, but the right answer for heavy or frequent builds.

On a plain Linux host, install the emulators once with docker run --privileged --rm tonistiigi/binfmt --install all.

The result under one tag

One tag, one image per arch.

Build several platforms under a single tag and you get a manifest list — one name that points at an image per architecture.

manifest list
rag-demo:0.1.0
one tag
amd64linux/amd64 image — for x86 hosts
arm64linux/arm64 image — for ARM hosts
A plain docker pull automatically selects the right image for the machine doing the pulling.

Hands-on · terminal

Let's prove it with a tiny image.

The demo image just prints the CPU architecture it runs on. Build it for each platform, run each one, and watch the reported arch change — same source, two binaries. Switch to the terminal.

buildx create build --platform run --platform

Terminal · build & run each arch

Same source, two architectures.

$ docker buildx build --platform linux/amd64 --load -t rag-demo:amd64 chapter_5/l4
$ docker run --rm --platform linux/amd64 rag-demo:amd64
architecture: x86_64
$ docker buildx build --platform linux/arm64 --load -t rag-demo:arm64 chapter_5/l4
$ docker run --rm --platform linux/arm64 rag-demo:arm64
architecture: aarch64
# same Dockerfile — the output proves each binary's arch

The AI catch

Multi-platform isn't free for heavy images.

A dependency-free demo builds in seconds on every arch. A real rag-ingestion image — torch, Docling — is a different story.

× what can go wrong
not every package ships wheels for every arch
missing wheel → the build fails on that arch
or falls back to a slow source compile under emulation
Worth knowing before you promise an ARM build.
✓ the commands are identical
same buildx build --platform recipe
just point at the real Dockerfile & context
verify wheels exist; prefer native builders for heavy images
The technique scales up — the dependencies are what you check.

Takeaway

The image is now portable.

One source, a builder that can target many platforms, and a manifest list that lets every host pull the image made for it.

But a multi-arch tag is a manifest list — and that only lives in a registry.
