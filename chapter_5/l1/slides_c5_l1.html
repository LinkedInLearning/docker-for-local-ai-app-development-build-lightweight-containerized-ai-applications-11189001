The gap

Working is not production.

The difference is a checklist — five things a production image gets right that a prototype usually doesn't.

A working image
Runs & passes tests

Boots on your laptop and in CI. Big, broad, unversioned — good enough to prove the idea.

A production image
Gets five things right

Small, secure, portable, versioned and distributed, and operable once it's running.

The checklist

Five columns to clear.

01

Size

Pull faster, store cheaper, expose less.

02

Security

Minimal surface, no secrets, least privilege.

03

Portability

Runs on the architecture you deploy to.

04

Distribution

Versioned and published to a registry.

05

Operability

Limits, health checks, graceful exit, observable.

Column 1 of 5 · Size

Smaller is faster and safer.

Pull faster. Small images deploy in seconds, not minutes.
Cost less to store. Less registry storage and bandwidth across every node.
Expose less. A multi-gigabyte image is full of things an attacker could use.

Column 2 of 5 · Security

Minimal surface, least privilege.

No secrets in layers. Credentials baked into the image leak with the image.
No known vulnerabilities. Patch the base and the dependencies that ship inside.
Least privilege. Don't run as root — drop down to an unprivileged user.

Column 3 of 5 · Portability

Built for where it runs.

Match the CPU architecture. The image has to run on the architecture you deploy to.
Laptop ≠ server. An image built only for your laptop's arch may not run on the server.
Build for the target. Produce the architectures your deploy environment actually uses.

Column 4 of 5 · Distribution

Versioned and published.

Versioned. Every build carries an explicit tag — not a moving :latest.
Published to a registry. So a deploy target can pull it from a known place.
Exactly the build you intend. The target runs the bits you tested — no surprises.

Column 5 of 5 · Operability

Behaves once it's running.

Resource limits. Cap CPU and memory so one container can't starve the host.
Health checks & graceful shutdown. Report readiness; drain cleanly on stop.
Observable. A way to see what it's doing while it's live.

Baseline first

You can't improve what you don't measure.

Before we fix anything, measure where the images stand today.

baseline the images 
$ docker images rag-ingestion rag-query     # how big are we starting?
$ docker history rag-ingestion:0.1.0        # which layers dominate?
$ dive rag-ingestion:0.1.0                   # wasted space, layer by layer
$ docker scout quickview rag-ingestion:0.1.0 # CVE count at a glance

The heavy ingestion image is large and has a wide surface. That's our starting point.
