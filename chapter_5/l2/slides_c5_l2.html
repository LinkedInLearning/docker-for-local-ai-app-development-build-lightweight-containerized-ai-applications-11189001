The problem

A single-stage image ships its build tools.

A normal image keeps everything the build needed. None of it runs in production — it's dead weight, and every extra package is more attack surface.

× single-stage image
compilers & build-essential
development headers
package & pip caches
the finished app on top
Everything the build touched is baked into the layers you ship.
✓ what actually runs
the installed dependencies
your application code
The runtime needs only the artifacts — not the machinery that produced them.

The idea

One stage to build, one stage to run.

The builder installs and compiles. The final, clean stage copies only the finished artifacts across — and ships those.

Builder stage
install build-essential
create a venv, pip install
compile native wheels
Holds the toolchain and the pip cache — then is thrown away.
COPY
--from=builder
Runtime stage
clean python:3.11-slim
copy the finished venv
copy the app code
No compilers, no caches — only what runs ends up in the image.

In practice

The two-stage Dockerfile.

Dockerfile_Ingestion
FROM python:3.11-slim AS builder
RUN apt-get update && apt-get install -y --no-install-recommends build-essential
RUN python -m venv /opt/venv
ENV PATH="/opt/venv/bin:$PATH"
COPY requirements-ingestion.txt .
RUN pip install -r requirements-ingestion.txt

FROM python:3.11-slim          # clean runtime — no compilers
COPY --from=builder /opt/venv /opt/venv
ENV PATH="/opt/venv/bin:$PATH"
COPY rag/ /app/rag/
What to notice
AS builderNames the first stage so the runtime stage can reach back into it.
venvDependencies install into an isolated /opt/venv — one tidy directory to move.
--from=builderCopies just the finished venv into the clean image.
thrown awayThe compiler toolchain and pip cache live only in the builder.

Supporting levers

Three levers that multiply the win.

Multi-stage is the big cut. These compound it.

1Smaller base

Start from slim instead of the full image — or a distroless base that ships no shell or package manager at all.

2Layer order

Put what changes least at the bottom — the cache idea from Chapter 3 — so rebuilds reuse the heavy dependency layers.

3.dockerignore

Keep junk out of the build context entirely — .venv, chroma_data/, and tests should never enter the build. Our repo had none until this chapter.

Terminal · build it both ways

Same dependencies, smaller image.

$ docker build -f chapter_5/l2/Dockerfile_Ingestion.singlestage -t rag-ingestion:single .
$ docker build -f chapter_5/l2/Dockerfile_Ingestion -t rag-ingestion:multi .
$ docker images rag-ingestion
REPOSITORY TAG IMAGE ID SIZE
rag-ingestion single a1b2c3d4e5f6 2.91GB
rag-ingestion multi f6e5d4c3b2a1 1.84GB

Same dependencies, smaller image — and it still does real work.

AI-specific note

Weights are data, not build output.

For our images the torch and Docling layers are the dramatic win. But be careful with model weights.

× baking weights into a layer
gigabytes frozen into the image
rebuild on every model swap
bloated, slow-to-pull layers
Weights aren't a build artifact — treating them like one defeats the slimming.
✓ handle weights at runtime
mount or fetch them at start
image stays lean and stable
swap models without rebuilding
Treat weights as data — we'll touch on this next lesson.

Takeaway

Build it, then throw the toolchain away.

Same dependencies, a smaller and safer image — because only the runtime artifacts make it into the final stage.
