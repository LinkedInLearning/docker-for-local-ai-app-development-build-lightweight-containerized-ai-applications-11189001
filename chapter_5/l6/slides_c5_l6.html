Going live · 1 of 4

Runtime guardrails.

A container with no limits can consume the whole host. In production you cap it and tell the platform how to handle failure.

bash
$ docker run --memory 1g --cpus 1.5 --restart unless-stopped \
    --read-only myuser/rag-query:0.1.0
--memory 1gCap RAM so one container can’t starve the host.
--cpus 1.5Bound CPU share — predictable, fair scheduling.
--restartunless-stopped — recover from crashes automatically.
--read-onlyImmutable filesystem shrinks the attack surface.

Going live · 2 of 4

Graceful shutdown & health.

Handle SIGTERM so in-flight work finishes when the platform stops the container, and add a HEALTHCHECK so the orchestrator knows when it’s actually ready — not just started. 

HEALTHCHECK --interval=30s --timeout=3s \
  CMD python -c "import urllib.request; urllib.request.urlopen('http://localhost:8080/health')" || exit 1
SIGTERM → gracefulCatch the stop signal, drain in-flight requests, then exit cleanly — no dropped work.
HEALTHCHECK → readyThe orchestrator routes traffic only when /health answers — started is not the same as ready.

Validate the artifact you ship.

The real production image, not a dev shell. Different from Chapter 4’s integration tests — those checked behavior; this checks the image you’re about to ship.

1
Run it

Start the production image exactly as it will run in deployment.

2
Hit /health

Confirm it comes up and reports healthy on its own.

3
Send a query

One real request end-to-end — the packaged app actually works.

4
Scan it

Re-scan the final image for CVEs before it leaves the build.

The point: you validate the exact bytes you ship — not a stand-in.

Going live · 4 of 4

Do it in CI, on every change.

All of this belongs in CI so it happens automatically, not by memory. A production pipeline is one ordered gate.

Build
build the image
Scan
scan for CVEs
Test
run the suite
Push
publish multi-arch
A failing scan or test stops the publish. Bad images never reach the registry.

In the pipeline

The key steps, in the workflow.

Wired into the project’s existing GitHub Actions workflow, build → scan → test → push looks like this.

# .github/workflows/main.yml (excerpt)
- run: docker build -f docker/Dockerfile_Query -t rag-query:${{ github.sha }} .
- run: docker scout cves --exit-code --only-severity critical,high rag-query:${{ github.sha }}
- run: pytest tests/
- run: docker buildx build --platform linux/amd64,linux/arm64 \
         -t myuser/rag-query:${{ github.ref_name }} --push docker/

--exit-code on the scan and a non-zero pytest both fail the job — so the buildx ... --push step only runs when the image is clean and the tests pass.

Beyond this course

Honest about scope.

KubernetesScheduling, self-healing, and rollouts across a cluster of machines.
