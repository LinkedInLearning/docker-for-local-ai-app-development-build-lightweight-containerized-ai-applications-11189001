Registries & image naming

Every image has an address.

Several registries exist — wherever the image goes, its name is structured, and the namespace is yours.

registry/namespace/repository:tag
registry
Where it lives. Defaults to Docker Hub (docker.io) when omitted.
namespace
Your account or org — the part that's yours to own.
repository
The image's name, e.g. rag-demo.
tag
Which version — 0.1.0, a SHA, or latest.
default
Docker Hub
docker.io — the registry used when none is named.
CI-native
GHCR
ghcr.io — ties into GitHub Actions.
cloud
ECR / GAR / ACR
Every cloud ships its own private registry.

Tagging strategy

Pin a version. Never trust latest.

The most important production habit: push an immutable version, because version tags are how you roll back with confidence.

× the moving tag
:latest
Points at whatever was pushed most recently
Silently changes under your feet
"Which build is in production?" → unanswerable
A moving target. You can't roll back to a tag that no longer means what it did.
✓ the immutable version
:0.1.0 · :git-sha
Each version is a fixed point
A 0.1.0 always means the same build
Rollback is just pull the old tag
An immutable identity. Provenance and confident rollbacks both depend on it.

What's happening
tagRe-label the local image under your namespace — no rebuild, just a new name for the same image.
pushUpload it to the registry under that name and tag.
dedupOnly layers the registry doesn't already have are uploaded — same caching idea as builds, over the network.

Multi-platform meets publishing

Build every platform and publish — one step.

This is the bridge from Lesson 4: buildx --push builds for each platform and publishes the manifest list at once — no intermediate --load.

terminal
# build amd64 + arm64, push as one tag
docker buildx build \
  --platform linux/amd64,linux/arm64 \
  -t myuser/rag-demo:0.1.0 \
  --push chapter_5/l4
why it matters
--platformTwo architectures built in the same invocation.
--pushSkips the local store entirely — a multi-arch image can't be --loaded, it goes straight to the registry.
manifestThe registry receives a manifest list: one tag fanning out to per-arch images.

Provenance & integrity

Prove it's the build you think it is.

For production you care not just about the tag, but about proving exactly which build is running.

Content digest
Every image has a @sha256:… digest that pins an exact build — unlike a tag, a digest can never move. Deploy by digest for an immutable target.
Signing & attestations
Tools like cosign or docker scout sign images and attach attestations, so consumers can verify the image came from you and hasn't been tampered with.

The AI note

Big images bite back at the registry.

With multi-gigabyte AI images, tag hygiene and layer caching matter more, not less.

Slow push / pull
Large layers take real time to upload and download — every deploy pays the transfer cost.
Storage limits
Multi-GB images consume registry storage fast; stale tags pile up if you don't prune.
Pull-rate limits
Registries throttle pulls — heavy images plus frequent deploys hit the ceiling sooner.

Takeaway

Built, hardened, portable — and published.

The image now lives in a registry, named by an immutable version, multi-arch, and ready for any deploy target to pull.
