The reality of building

Requirements change — constantly.

We start from a set of requirements, but they move: add a library, bump a version, pull in a new model, restructure the project. During a prototype, this happens many times a day.

Every change means a rebuild. So the real question is: how do we structure the image so rebuilds stay fast as the requirements keep moving?

Recall · from Chapter 2

Lean on the build cache.

We covered this in Chapter 2, so just the reminder: Docker builds layers top to bottom and reuses each cached layer until one of its inputs changes — then that layer, and every layer below it, rebuilds.

That hands us one lever for fast rebuilds: where a change lives in the Dockerfile decides how much gets rebuilt — so stable things go on top, volatile things on the bottom.

Our dev image

Three tiers of stability.

Order the Dockerfile so the further down a change lives, the cheaper the rebuild.

Looking ahead · Lesson 5

If the base never changes, why rebuild it?

Split the work into two images — build the stable tier once, rebuild only the thin layer on top.

Base image · built once
rag-base:0.1
Heavy, stable tooling — OS, runtime, system packages. Rarely rebuilt.
FROM rag-base:0.1
Dev image · volatile
rag-dev:0.1
Only the project-specific pieces. This is the thin layer we rebuild all day.

Takeaway

Stable on top, volatile on the bottom.

Designing for change isn't premature optimization — it's the difference between a rebuild you wait through and one you don't notice.

Over to VS Code — the RAG project's two Dockerfiles are this exact split: a stable base, and a volatile dev image built on top.

  docker/Dockerfile_Base foundation · built once
FROM ubuntu:22.04
ARG QUARTO_VER="1.7.32"
RUN bash install_dependencies.sh
RUN bash install_quarto.sh
# OS + system deps + Quarto — rarely changes
  docker/Dockerfile_Dev volatile · rebuilt all day
FROM rkrispin/python-base:0.0.4
ARG PYTHON_VER  VENV_NAME  RUFF_VER
RUN bash install_uv.sh …
RUN python cache_docling_models.py
# Python env + deps + model cache — moves often

Base carries the heavy, stable tooling and is built once; Dev is FROM the base and holds only what changes as we prototype.
