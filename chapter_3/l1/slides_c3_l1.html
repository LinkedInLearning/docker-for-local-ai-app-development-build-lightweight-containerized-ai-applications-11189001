The reality of building

Requirements change — constantly.

We start from a set of requirements, but they move: add a library, bump a version, pull in a new model, restructure the project. During a prototype, this happens many times a day.

Every change means a rebuild. So the real question is: how do we structure the image so rebuilds stay fast as the requirements keep moving?

The build cache

Change one layer, rebuild everything below.

FROM python:3.11-slim base image cached
RUN apt-get install … system setup cached
RUN pip install … dependencies cached
COPY . . project code rebuilt
Docker walks top → bottom.

It reuses each layer until an input changes.

The first changed instruction — and every layer after it — rebuilds from scratch.

The principle

Stable on top, volatile on the bottom.

Dependencies change rarely; code changes constantly. Order the instructions to match.

Fragile 
# code before deps
COPY . .
RUN pip install -r requirements.txt
Any code edit invalidates COPY . . — so pip install re-runs every time.
Cache-friendly 
# deps first, code last
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
pip install re-runs only when deps change. Code edits rebuild one cheap layer.

Our dev image

Three tiers of stability.

Order the Dockerfile so the further down a change lives, the cheaper the rebuild.

Looking ahead · Lesson 5

If the base never changes, why rebuild it?

Split the work into two images — build the stable tier once, rebuild only the thin layer on top.

Base image · built once
rag-base:0.1
Heavy, stable tooling — OS, runtime, system packages. Rarely rebuilt.
FROM rag-base:0.1
Dev image · volatile
rag-dev:0.1
Only the project-specific pieces. This is the thin layer we rebuild all day.

Takeaway

Stable on top, volatile on the bottom.

Designing for change isn't premature optimization — it's the difference between a rebuild you wait through and one you don't notice.
