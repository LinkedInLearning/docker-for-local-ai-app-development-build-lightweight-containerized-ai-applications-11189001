Best practice · 1 of 4

Version your images with meaningful tags.

An image without a version is a moving target — latest today is not latest next week. Bump the tag deliberately when the environment changes. The tag becomes a pin: a teammate, a CI job, or your future self can pull that exact environment back.

python-dev-rag-docker:0.0.3
A new dependency or a base-image change earns a new tag. Pull 0.0.3 and you get exactly what you had.
2
Best practice · 2 of 4

Script your builds.

Build commands grow long — platforms, build args, tags, a push at the end. Put them in a shell script so the script is the documentation of how the image is built. 

# docker/build_dev_docker.sh
docker buildx build . -f Dockerfile_Dev \
  --platform linux/amd64,linux/arm64 \
  --build-arg PYTHON_VER=3.11 \
  --build-arg VENV_NAME=python-3.11-dev \
  -t rkrispin/python-dev-rag-docker:0.0.3

The image settings — name, tag, versions — live at the top as variables, so bumping a version is a one-line edit. Anyone can rebuild the exact image without knowing the incantation.

Best practice · 3 of 4

Split a base image from a dev image.

Remember the stability tiers: the OS and toolchain almost never change; the project's dependencies and code change constantly. Put each in its own image.

ubuntu:22.04
The starting point.
Base imagestable
System deps, Quarto, shell setup. Built once.

Dev container
Where you actually work.

Practice 3 · the base image

Stable, project-agnostic foundation.

In our repo that's docker/Dockerfile_Base: Ubuntu, system dependencies, Quarto, the shell. Built once, tagged python-base:0.0.4, pushed to a registry. It rarely changes. 

# docker/Dockerfile_Base
FROM ubuntu:22.04
RUN bash ./settings/install_dependencies.sh
RUN bash ./settings/install_quarto.sh $QUARTO_VER

The heavy, slow-changing work — the OS layer and system tooling — happens here, once.

Practice 3 · the dev image

Thin layer of project-specific pieces.

docker/Dockerfile_Dev
FROM rkrispin/python-base:0.0.4
COPY install_uv.sh requirements.txt \
     settings/
RUN bash ./settings/install_uv.sh \
    $VENV_NAME $PYTHON_VER $RUFF_VER
Builds FROM the base and adds only uv, requirements.txt, models and tooling.
Why it matters

The base image is already built. The dev image sits on top of it — a few project-specific layers, nothing more.

The win: change a dependency → rebuild only the thin dev image. You never reinstall the OS and system tools just to add a Python package. The expensive, stable work is done once.

Best practice · 4 of 4

Start new projects from a template.

Once you've built a good containerized setup, don't recreate it from scratch every time. Capture it as a GitHub template repository — a new project starts as a copy, already wired for containerized development.

Template repo captures

  • Dockerfile_Base / Dockerfile_Dev
  • build_*.sh scripts
  • docker-compose.yml
  • .devcontainer/devcontainer.json
