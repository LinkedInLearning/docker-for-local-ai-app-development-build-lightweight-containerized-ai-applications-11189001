Step 1 · identify the services

Three core services in our RAG system.

Before deciding container boundaries, name the services. For our running example, three boxes:

Pipeline

Ingestion pipeline

Processes documents and stores their embeddings.

Database

Vector database

Stores embeddings, enables fast similarity search.

API

Query pipeline

Receives requests, retrieves context, calls the LLM, returns response.

Now we can ask the design question: should these run in one container, or several?

Approach 1

When one container is enough.

For some applications, putting both pipelines in a single container is the right call.

! prototype-style

Single container

Ingestion + query running together as one workflow.

one container
ingestion pipeline
vector store (in-memory)
query pipeline
  • Simpler to reason about
  • One image, one process, one log stream
  • No service-to-service networking
→ example

Session-based RAG

A user uploads documents during a session and immediately asks questions about them.

  • Documents are processed temporarily
  • Used during the session, discarded afterward
  • No long-lived knowledge base to maintain
When this fits: the two pipelines operate as a single workflow. Splitting them adds complexity without benefit.

Approach 2

When separation pays off.

In most production setups, splitting ingestion and query into independent containers becomes useful. Components that operate independently are good candidates for separate containers.

✓ production-style

Separate containers

Each service in its own container, communicating over the network.

container 1
ingestion
container 2
vector store
container 3
query API

Each can be built, deployed, and scaled on its own.

→ reasons to separate
  • Different cadences — ingestion runs occasionally; query runs continuously.
  • Different resources — one is CPU-heavy, the other is latency-sensitive.
  • Independent scaling — 10 query workers, 1 ingestion worker.
  • Independent deployment — ship query updates without touching ingestion.
  • Failure isolation — a stuck job won't take down the API.

The order matters

Services. Requirements. Then containers.

It's tempting to jump straight to the container diagram. Don't. Without clear services and requirements, the container choices won't survive contact with reality.

1
Identify services What does the application actually do, end to end?
2
Define requirements What does each service need? Runtime, libs, env, ports, resources.
3
Decide containers How to package those services for the target environment.

The goal isn't "make it run." It's predictable, consistent deployment across environments.
