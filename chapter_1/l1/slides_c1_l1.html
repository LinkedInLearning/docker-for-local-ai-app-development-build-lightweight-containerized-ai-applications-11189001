The classic problem

Every developer has heard this sentence.

Anyone who has shipped software has shipped this.

“

It works on my machine…

— every developer, eventually

Works on the laptop that wrote it. Breaks on a teammate's, on CI, in staging, in prod.

Why it breaks

Same code, different machines, different reality.

Tiny differences in runtime, libraries, OS, or architecture are enough to make the same code behave differently.

It gets worse for AI

AI applications have many moving parts.

Traditional apps already face this. AI apps multiply the number of pieces that have to line up.

Python & librariesSpecific runtime, dozens of pinned packages, model dependencies.
External APIs & LLMsOpenAI, Gemini, Anthropic — keys, rate limits, version changes.
Vector databasesChroma, Qdrant, Postgres+pgvector — each with their own quirks.
Ingestion pipelinesDocument parsing, chunking, embedding — multi-step batch jobs.
GPU & system libsCUDA, cuDNN, glibc, OpenSSL versions that all have to align.
Backend & web frameworksFastAPI, Flask, Streamlit — and the runtime they all need.

“Install Python 3.12, these 25 libraries, a vector database, download this model, configure these environment variables… and hopefully everything works.”

The fix

Ship the environment, not the instructions.

Containers package the application with its dependencies and system configuration into a reproducible environment. Instead of describing the setup, you ship it.

Container
Python  3.12.11
Pandas  2.2.2
OS      Ubuntu 22.04
+ libs & env
macOSsame env
Windowssame env
Linuxsame env

If it works locally, it should work elsewhere. If it fails elsewhere, you can reproduce the failure locally.

The whole lifecycle

Same environment, end to end.

Once the application is in a container, the same environment runs at every stage of its life.

The real benefit isn't "easier to run." It's reproducibility across the entire lifecycle.

What we'll do in this course

Containerize a real AI application, end to end.

We'll use a RAG system as the running example. The same techniques apply to agents, forecasters, recommenders — any AI application with more than one moving part.

Takeaway

Reproducibility is the point.

Not "easier to run" — same behavior across dev, test, and production.

Ship the environment, not the instructions.
